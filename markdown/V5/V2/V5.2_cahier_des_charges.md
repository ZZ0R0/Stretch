# V5.2 — Cahier des charges

> Spécification des exigences pour la V5.2 de Stretch.
> Deux axes : **Performance** (GPU) et **Résultats** (apprentissage).

---

## §1 — Contexte et motivation

### 1.1 Héritage V5.0 / V5.1

La V5.0 a prouvé l'apprentissage réel du réseau (Δ = +84 pp en mode inversé). La V5.1 a complété la visualisation, mesuré le RandomBaseline (68.8%) et le Remap (50.0%), et documenté l'architecture.

### 1.2 Problèmes identifiés

1. **Saturation du reward** : en mode normal, $r(t) \approx +1$ constant → la dopamine ne porte pas d'information → renforcement aveugle → l'apprentissage dégrade la topologie (−9.8 pp).

2. **Re-learning trop lent** : Remap donne 50.0% global. L'homéostasie ($\rho = 0.0001$) est insuffisante pour effacer les routes pré-renforcées en 5000 ticks.

3. **Divergence CPU/GPU** : les fonctionnalités V5 sustained (decay adaptatif, réverbération, reset policy) n'existent que sur CPU. Le GPU tourne en V4 pur. Les résultats V5 n'ont jamais bénéficié de ces mécanismes.

4. **readout hardcodé** : le shader `readout.wgsl` a `num_classes = 2` en dur → bloque l'extension à >2 classes.

### 1.3 Objectif V5.2

Résoudre la saturation du reward (RPE) et unifier le pipeline CPU/GPU pour que **toutes les fonctionnalités V5+ tournent sur GPU** avec des performances au moins égales à la V5.1.

## §2 — Exigences fonctionnelles

### §2.1 — RPE (Reward Prediction Error)

| ID | Exigence | Priorité |
|---|---|---|
| RPE-1 | Maintenir une moyenne glissante exponentielle $\bar{r}(t)$ du reward | **P0** |
| RPE-2 | Le signal de plasticité utilise $\delta(t) = r(t) - \bar{r}(t)$ au lieu de $r(t)$ brut | **P0** |
| RPE-3 | Le paramètre $\alpha$ (taux de mise à jour de $\bar{r}$) est configurable dans `[reward]` | **P0** |
| RPE-4 | $\alpha$ par défaut = 0.05, plage valide $[0.001, 0.5]$ | P1 |
| RPE-5 | Le RPE est actif sur les deux backends (CPU et GPU) | **P0** |
| RPE-6 | Exposer $\bar{r}(t)$ et $\delta(t)$ dans les métriques par tick | P1 |

### §2.2 — Port V5 Sustained → GPU

| ID | Exigence | Priorité |
|---|---|---|
| SUS-1 | Le decay adaptatif V5 tourne sur GPU via un shader dédié | **P0** |
| SUS-2 | La réverbération locale tourne sur GPU avec un buffer `prev_activations` | **P0** |
| SUS-3 | Le reset policy V5 ("full" / "partial" / "none") fonctionne sur GPU | **P0** |
| SUS-4 | Les paramètres V5 sustained (k_local, reverb_gain, reset_policy) sont transmis au GPU via GpuParams | **P0** |
| SUS-5 | Le résultat GPU doit être numériquement identique (±ε floating point) au résultat CPU pour les mêmes paramètres | P1 |

### §2.3 — Corrections GPU

| ID | Exigence | Priorité |
|---|---|---|
| GPU-1 | `readout.wgsl` lit `num_classes` depuis GpuParams au lieu de la constante `2u` | **P0** |
| GPU-2 | GpuParams inclut les champs V5.2 : `reverb_gain`, `k_local`, `reset_policy`, `rpe_alpha`, `reward_baseline`, `rpe_delta` | **P0** |
| GPU-3 | Le pipeline GPU ajoute 2 nouvelles passes : `adaptive_decay` et `reverberation` (insérées entre propagation et plasticity) | **P0** |
| GPU-4 | L'ordre des passes GPU V5.2 est : reset → inject → zones → source_contribs → propagation → apply_influences → **reverberation** → **adaptive_decay** → dissipation → plasticity → budget → sync → readout → metrics | P1 |

### §2.4 — Oubli accéléré (conditionnel)

| ID | Exigence | Priorité |
|---|---|---|
| OBL-1 | Mode oubli accéléré : $\rho_{\text{eff}} = \rho_0 + \rho_{\text{boost}} \cdot \max(0, -\delta(t))$ | P1 |
| OBL-2 | $\rho_{\text{boost}}$ configurable, par défaut = 0.01 | P1 |
| OBL-3 | Actif uniquement quand `v5_task.task_mode = Remap` ou quand activé explicitement | P1 |
| OBL-4 | Implémenté sur les deux backends | P1 |

### §2.5 — Modulation par la marge

| ID | Exigence | Priorité |
|---|---|---|
| MAR-1 | Le reward effectif est pondéré : $r_{\text{eff}} = r \cdot \frac{1}{1 + \beta_M \cdot |M|}$ | P2 |
| MAR-2 | $\beta_M$ configurable, par défaut = 0.1 | P2 |
| MAR-3 | La marge $M$ est déjà calculée par le readout — pas de nouveau calcul requis | P2 |

### §2.6 — Multi-seed et reproductibilité

| ID | Exigence | Priorité |
|---|---|---|
| SEED-1 | Option CLI `--seeds 42,123,456,789,1337` pour lancer N runs consécutifs | P1 |
| SEED-2 | Résultats rapportés en $\mu \pm \sigma$ | P1 |
| SEED-3 | Export d'un CSV récapitulatif par condition | P1 |

## §3 — Exigences non-fonctionnelles

### §3.1 — Performance

| ID | Exigence | Priorité |
|---|---|---|
| PERF-1 | Le pipeline V5.2 GPU n'ajoute pas plus de 15% d'overhead par tick par rapport au V4.2 GPU | **P0** |
| PERF-2 | Le temps de tick GPU à 50k nœuds / 500k arêtes reste < 5 ms | P1 |
| PERF-3 | La mémoire GPU supplémentaire (buffer prev_activations) = 4B × N nœuds ≤ 200 KB à 50k | P1 |
| PERF-4 | Aucun readback CPU-GPU intermédiaire ajouté (le RPE est calculé après le readout, côté CPU) | **P0** |

### §3.2 — Qualité

| ID | Exigence |
|---|---|
| QUAL-1 | 0 erreur, 0 warning sur les 3 crates (`cargo build --release`) |
| QUAL-2 | Les shaders WGSL sont validés par `naga` sans erreur |
| QUAL-3 | La documentation V5.2 est complète avant implémentation |

### §3.3 — Compatibilité

| ID | Exigence |
|---|---|
| COMPAT-1 | Les configs V5.0/V5.1 existantes (7 fichiers config_v5_*.toml) restent fonctionnelles sans modification |
| COMPAT-2 | Le comportement par défaut (sans champs V5.2) est identique à V5.1 |
| COMPAT-3 | Le backend CPU reste pleinement fonctionnel (fallback si pas de GPU) |

## §4 — Critères d'acceptation

### §4.1 — Matrice de résultats cible

La V5.2 doit produire la matrice suivante (Symmetric, 50k nœuds, 10k ticks) :

| | TopologyOnly | FullLearning | Δ (FL − Topo) | Gate |
|---|---|---|---|---|
| **Normal** [0→0, 1→1] | ~97% | **≥ 97%** | **≥ 0 pp** | ✅ requis |
| **Inversé** [0→1, 1→0] | ~3% | **≥ 80%** | **≥ +77 pp** | ✅ requis |

### §4.2 — Remap

| | Phase 1 (0–5k) | Phase 2 (5k–10k) | Gate |
|---|---|---|---|
| **Remap** | > 90% | **> 70%** | ✅ requis |

### §4.3 — Performance GPU

| Métrique | Cible | Gate |
|---|---|---|
| Overhead V5.2 vs V4.2 | < 15% | ✅ requis |
| Tick time (50k) | < 5 ms | P1 |
| Mémoire GPU ajoutée | < 1 MB | P1 |

### §4.4 — Cohérence mathématique

| Métrique | Cible |
|---|---|
| CPU ≡ GPU (mêmes params, mêmes résultats ±0.1%) | ✅ requis |
| RPE : $\bar{r}$ converge en mode normal vers ~1.0 | ✅ requis |
| RPE : $\delta$ ≈ 0 en régime permanent normal | ✅ requis |
| RPE : $\delta$ fortement négatif après erreur en mode normal | ✅ requis |

## §6 — Hors périmètre V5.2

Les éléments suivants sont explicitement exclus du périmètre V5.2 :

- Scaling à >50k nœuds (reporté à V5.3, conditionné au succès du RPE)
- Hiérarchie micro/méso/macro (V6)
- Tâches à >2 classes (activation seulement si GPU-1 résolu)
- Dopamine spatiale sur GPU (spatial_lambda = 0 en V5)
- Modifications de la topologie (knn_3d, radius_3d — stables)
- Refactoring de stretch-viz (stable depuis V5.1)

## §7 — Dépendances d'implémentation

```
RPE-1,2,3 ─────────────── pas de dépendance (peut commencer immédiatement)
     │
     ├──→ RPE-5 (port GPU : passer δ via GpuParams → GPU-2)
     │
     ├──→ OBL-1 (dépend de δ(t) calculé par RPE)
     │
     └──→ MAR-1 (dépend du readout margin)

SUS-1,2,3 ──→ GPU-2 (GpuParams doit avoir les champs V5)
     │                ──→ GPU-3 (nouveaux pipelines)
     │                       ──→ GPU-4 (ordre des passes)
     │
     └──→ SUS-4 (upload GpuParams étendu)

GPU-1 ─────────────────── pas de dépendance (fix trivial)

SEED-1,2,3 ────────────── pas de dépendance (CLI only)
```

**Ordre recommandé** :

1. GPU-1 (fix readout) + RPE-1,2,3,4 (RPE côté CPU)
2. GPU-2 (GpuParams étendu) + RPE-5 (RPE → GpuParams)
3. SUS-1,2,3,4 (port sustained → GPU) + GPU-3,4 (nouvelles passes)
4. Tests matrice 2×2 → validation gate §4.1
5. OBL-1,2,3,4 (oubli accéléré, si Remap échoue)
6. SEED-1,2,3 (multi-seed)
7. MAR-1,2,3 (marge, si variance élevée)
