# V5.orch — Risques et gates : Framework d'exécution unifié

> Analyse des risques du refactoring et critères de passage vers V5.2.

---

## 1. Matrice des risques

### RO1 — Régression subtile par changement d'ordre d'exécution

| Attribut | Valeur |
|---|---|
| **Probabilité** | Moyenne |
| **Impact** | Élevé — les résultats changent silencieusement |
| **Mécanisme** | En déplaçant le code de `step_cpu()`/`step_gpu()` vers les backends, une phase pourrait être exécutée dans un ordre légèrement différent (ex : dopamine decay avant vs après le pipeline). Cela changerait le timing de la plasticité. |
| **Détection** | Protocole O1 (CPU bit-identical) détecte toute divergence |
| **Mitigation** | Migration étape par étape (6 étapes dans le schéma de migration). Validation non-régression après chaque étape. |
| **Fallback** | Git revert de l'étape fautive. Analyse diff. |

### RO2 — Champ oublié dans TickParams → GpuParams

| Attribut | Valeur |
|---|---|
| **Probabilité** | Moyenne (GpuParams a 45+ champs) |
| **Impact** | Modéré — le GPU utilise une valeur par défaut (0.0) au lieu du paramètre réel |
| **Mécanisme** | Le mapping `TickParams → GpuParams` dans `GpuBackend::build_gpu_params()` oublie un champ. Le struct GpuParams est initialisé partiellement, le champ oublié vaut 0. |
| **Détection** | Protocole O2 (GPU non-régression) détecte la divergence. Le compilateur Rust aide si on utilise la syntaxe `GpuParams { field: ..., }` sans `..Default::default()`. |
| **Mitigation** | **Ne PAS utiliser `..Default::default()`** dans le constructeur de GpuParams. Forcer l'initialisation explicite de tous les champs → le compilateur signale tout champ manquant. |
| **Fallback** | Diff du mapping V5.1 (step_gpu) vs V5.orch (build_gpu_params) |

### RO3 — Buffer CPU non transféré au CpuBackend

| Attribut | Valeur |
|---|---|
| **Probabilité** | Faible |
| **Impact** | Élevé — crash ou résultat incorrect |
| **Mécanisme** | Les buffers temporaires (`buf_source_contribs`, `buf_fired_list`, etc.) sont des `Vec` pré-alloués dans `Simulation`. Si le `CpuBackend` ne les possède pas ou n'y a pas accès, le code crash ou alloue à chaque tick. |
| **Détection** | Crash immédiat, ou protocole O4 détecte un overhead anormal |
| **Mitigation** | Les buffers sont déplacés dans `CpuBackend` lors de l'étape 5. Le `CpuBackend` les pré-alloue à l'init. |
| **Fallback** | Passer les buffers par référence mutable au lieu de les encapsuler |

### RO4 — La viz ne reçoit plus les snapshots correctement

| Attribut | Valeur |
|---|---|
| **Probabilité** | Faible |
| **Impact** | Modéré — la visualisation ne fonctionne plus |
| **Mécanisme** | Le `VizSnapshot` est construit dans `step()` après l'exécution du tick. Si l'interface publique de `Simulation` change (ex : `domain` non accessible), la viz ne peut plus lire l'état. |
| **Détection** | Protocole O9 (viz test) |
| **Mitigation** | Conservation de l'interface publique de `Simulation`. Le `Domain` et les métriques restent accessibles. Seule la mécanique interne de `step()` change. |
| **Fallback** | Ajouter un getter `Simulation::domain()` si nécessaire |

### RO5 — Le refactoring prend trop de temps et retarde V5.2

| Attribut | Valeur |
|---|---|
| **Probabilité** | Faible (le refactoring est mécanique, ~500-700 lignes) |
| **Impact** | Modéré — retard sur V5.2 |
| **Mécanisme** | La migration étape par étape révèle des dépendances inattendues. Ex : le `CpuBackend` a besoin d'accéder à `ZoneManager` qui est dans `Simulation`. Le passage de références mutable crée des conflits d'emprunt (borrow checker Rust). |
| **Détection** | Si l'étape 5 (CpuBackend) bloque sur le borrow checker |
| **Mitigation** | **Design pragmatique** : le `CpuBackend` peut recevoir les données par référence mutable dans `execute_tick()` au lieu de posséder tous les sous-systèmes. Alternative : passer un `&mut SimState` qui regroupe domain + zones + buffers. |
| **Fallback** | Simplifier : le CpuBackend est une closure dans `Simulation` qui capture l'état par référence, au lieu d'un struct séparé. Perd la symétrie trait mais résout les problèmes d'ownership. |

## 4. Risque global

**Confiance** : **Très haute** (9/10).

Le refactoring V5.orch est **purement mécanique** :
- Pas de nouvel algorithme
- Pas de nouvelle fonctionnalité
- Pas de modification de shader
- Pas de changement de comportement observable

Les risques sont principalement liés à l'artisanat du refactoring (oubli de champ, ordre d'exécution). Le borrow checker Rust et les tests de non-régression forment un double filet de sécurité.

Le seul risque non-trivial est RO5 (borrow checker), mitigé par un design pragmatique (passage par &mut au lieu de possession).
