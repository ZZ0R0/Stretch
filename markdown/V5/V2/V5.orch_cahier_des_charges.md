# V5.orch — Cahier des charges : Framework d'exécution unifié

> Spécification des exigences pour le refactoring de l'orchestrateur Stretch.
> Ce document est un **pré-requis** à la V5.2.

---

## §1 — Contexte et motivation

### 1.1 Problème actuel

La simulation Stretch a deux chemins d'exécution complètement séparés :

- **`step_gpu()`** (~160 lignes) : construit `GpuParams` (45+ champs), soumet le pipeline GPU en single submit, puis gère le readout, le reward, la décision et la dopamine côté CPU.
- **`step_cpu()`** (~200 lignes) : exécute toutes les phases séquentiellement (zones, stimuli, propagation, réverbération, dissipation, plasticité, readout, reward, dopamine).

Ces deux fonctions **dupliquent** la logique métier suivante :

| Logique dupliquée | step_gpu() | step_cpu() | Lignes dupliquées |
|---|---|---|---|
| Gestion du trial (start, stimulus, readout timing) | ✅ | ✅ | ~25 |
| Readout → decision (argmax) | ✅ | ✅ | ~15 |
| Reward → dopamine update | ✅ | ✅ | ~25 |
| Spatial dopamine (reward center) | ✅ | ✅ | ~20 |
| Dopamine decay | ✅ | ✅ | ~5 |
| reward_system.clear() | ✅ | ✅ | ~2 |
| **Total** | | | **~92 lignes** |

### 1.2 Conséquences

1. **Divergence silencieuse** : les fonctionnalités V5 sustained (decay adaptatif, réverbération, reset policy) n'existent que dans `step_cpu()`. Le GPU tourne en V4 pur. Aucune erreur de compilation, aucun avertissement.
2. **Double implémentation** : chaque nouvelle fonctionnalité (RPE, oubli accéléré, marge) doit être implémentée **deux fois** avec le risque de divergence.
3. **Difficulté de test** : valider la cohérence CPU/GPU requiert des tests manuels comparatifs non automatisables.
4. **Complexité croissante** : à chaque version, la duplication s'aggrave. V5.2 ajouterait ~80 lignes dupliquées supplémentaires.

### 1.3 Objectif

Créer un **framework d'exécution unifié** où :
- L'**orchestrateur** définit la séquence logique du tick (phases, trial, reward, dopamine) — écrit **une seule fois**.
- Le **backend** (CPU ou GPU) exécute les phases de calcul — seule la mécanique de calcul diffère.
- L'ajout d'une fonctionnalité à l'orchestrateur la rend automatiquement disponible sur les deux backends.

## §2 — Exigences fonctionnelles

### §2.1 — Architecture

| ID | Exigence | Priorité |
|---|---|---|
| ORCH-1 | Définir un `enum Phase` exhaustif listant toutes les phases de calcul d'un tick | **P0** |
| ORCH-2 | Définir un `struct TickParams` consolidant tous les paramètres par tick (remplace la construction ad-hoc de GpuParams) | **P0** |
| ORCH-3 | Définir un `trait ComputeBackend` avec les méthodes d'exécution des phases | **P0** |
| ORCH-4 | Implémenter `GpuBackend` qui encapsule `GpuContext` existant et implémente `ComputeBackend` | **P0** |
| ORCH-5 | Implémenter `CpuBackend` qui encapsule la logique CPU existante et implémente `ComputeBackend` | **P0** |
| ORCH-6 | Refactorer `Simulation::step()` en **orchestrateur unique** qui délègue au backend via le trait | **P0** |
| ORCH-7 | Le dispatch CPU/GPU est transparent pour la logique métier (trial, reward, dopamine) | **P0** |

### §2.2 — Séquence logique

| ID | Exigence | Priorité |
|---|---|---|
| SEQ-1 | L'orchestrateur gère la séquence de phases dans l'ordre canonique défini par `Phase` | **P0** |
| SEQ-2 | La gestion des trials (timing, stimulus_class, need_readout) est faite **une seule fois** dans l'orchestrateur | **P0** |
| SEQ-3 | Le readout est exécuté par le backend, mais la décision (argmax) est faite dans l'orchestrateur | **P0** |
| SEQ-4 | Le reward, la dopamine, et le spatial dopamine sont gérés dans l'orchestrateur | **P0** |
| SEQ-5 | Les métriques et le perf monitoring restent dans l'orchestrateur | P1 |

### §2.3 — Extensibilité

| ID | Exigence | Priorité |
|---|---|---|
| EXT-1 | Ajouter une phase au pipeline ne requiert que : (a) un variant dans `Phase`, (b) son implémentation dans chaque backend | **P0** |
| EXT-2 | Le compilateur Rust (exhaustive match) doit forcer l'implémentation de chaque nouvelle phase dans chaque backend | **P0** |
| EXT-3 | `TickParams` est extensible : les nouveaux champs V5.2+ (RPE, sustained, oubli) s'y ajoutent naturellement | P1 |

### §2.4 — Backends

| ID | Exigence | Priorité |
|---|---|---|
| GPU-B1 | `GpuBackend` conserve le pattern single CommandEncoder + single submit + zéro readback intermédiaire | **P0** |
| GPU-B2 | `GpuBackend` accepte un `TickParams` et le convertit en `GpuParams` sans perte | **P0** |
| GPU-B3 | `GpuBackend` exécute toutes les phases dans un seul `run_full_tick()` — pas de submit par phase | **P0** |
| CPU-B1 | `CpuBackend` conserve les optimisations : fired-only (C1), hot-edges (C3), incremental budget (C4), node_needs_update (C5) | **P0** |
| CPU-B2 | `CpuBackend` conserve le parallélisme rayon | **P0** |
| CPU-B3 | `CpuBackend` encapsule les buffers temporaires (source_contribs, fired_list, influences, prev_activations) | P1 |

## §3 — Exigences non-fonctionnelles

### §3.1 — Performance

| ID | Exigence | Priorité |
|---|---|---|
| PERF-O1 | Le refactoring n'ajoute **aucun overhead mesurable** sur le chemin GPU (même single submit) | **P0** |
| PERF-O2 | Le refactoring n'ajoute pas plus de **5% d'overhead** sur le chemin CPU | **P0** |
| PERF-O3 | Aucune allocation supplémentaire par tick (les buffers sont réutilisés) | P1 |

### §3.2 — Qualité

| ID | Exigence |
|---|---|
| QUAL-O1 | 0 erreur, 0 warning (`cargo build --release`) |
| QUAL-O2 | Non-régression totale : résultats identiques bit-à-bit en CPU, identiques ±ε en GPU |
| QUAL-O3 | Le code dupliqué entre step_gpu() et step_cpu() est réduit à **≤10 lignes** (vs ~92 actuellement) |

### §3.3 — Compatibilité

| ID | Exigence |
|---|---|
| COMPAT-O1 | Toutes les configs existantes (V4, V5.0, V5.1) restent fonctionnelles sans modification |
| COMPAT-O2 | Le CLI et stretch-viz ne nécessitent aucune modification d'interface |
| COMPAT-O3 | Les diagnostics (SustainTracker, path tracer) continuent de fonctionner en mode CPU |

|---|---|---|---|

| Lignes dupliquées trial/reward/dopa | ~92 | **≤ 10** | ✅ requis |

### §4.3 — Extensibilité vérifiée

## §5 — Hors périmètre

Les éléments suivants sont explicitement exclus du périmètre V5.orch :

- Implémentation du RPE (V5.2a)
- Port des fonctionnalités V5 sustained sur GPU (V5.2b)
- Nouveaux shaders WGSL (V5.2b)
- Oubli accéléré, marge, multi-seed (V5.2c)
- Modification de stretch-viz ou stretch-cli
- Refactoring de gpu.rs au-delà de l'encapsulation dans GpuBackend

Le périmètre V5.orch est **purement structurel** : réorganiser le code existant sans changer la fonctionnalité.

## §6 — Livrables

| Livrable | Description |
|---|---|
| `Phase` enum | Défini dans un nouveau module ou dans `simulation.rs` |
| `TickParams` struct | Paramètres unifiés du tick |
| `ComputeBackend` trait | Interface commune CPU/GPU |
| `GpuBackend` struct | Wrapper autour de `GpuContext` |
| `CpuBackend` struct | Encapsulation de la logique CPU |
| Orchestrateur refactoré | `Simulation::step()` unique, sans `step_gpu()`/`step_cpu()` |
| Tests de non-régression | Validation matrice 2×2 identique à V5.1 |
