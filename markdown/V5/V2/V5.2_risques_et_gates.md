# V5.2 — Risques et gates

> Analyse des risques de la V5.2 et critères de passage entre phases.

---

## 1. Matrice des risques

### R1 — RPE trop conservateur (δ ≈ 0 en permanence)

| Attribut | Valeur |
|---|---|
| **Probabilité** | Moyenne |
| **Impact** | Élevé — l'apprentissage s'arrête complètement |
| **Mécanisme** | Si α est trop petit, $\bar{r}$ suit $r$ de très près → $\delta \approx 0$ → aucune plasticité. Si α est trop grand, $\bar{r}$ oscille trop → signal bruité mais informationnel. |
| **Détection** | Protocole 1.1 : vérifier que $\text{Var}(\delta) > 0.01$ sur les 1000 derniers ticks |
| **Mitigation** | Balayage de α (protocole 1.3). Si aucun α ne fonctionne : ajouter un plancher $\|\delta\| \geq \epsilon$ (forcer un signal minimum). |
| **Fallback** | Désactiver RPE (`rpe_enabled = false`) → retour au comportement V5.1 |

### R2 — L'oubli accéléré détruit les routes en mode normal

| Attribut | Valeur |
|---|---|
| **Probabilité** | Faible (le RPE protège : δ ≈ 0 en régime normal → ρ_eff = ρ₀) |
| **Impact** | Élevé — l'accuracy normale chute |
| **Mécanisme** | En mode normal, les erreurs rares donnent δ < 0 → ρ_eff augmente → les arêtes utiles sont affaiblies. Mais ces événements sont rares (2% des trials) et transitoires. |
| **Détection** | Protocole 1.2 : Δ_normal < 0 après activation de l'oubli accéléré |
| **Mitigation** | Conditionner l'oubli accéléré à `task_mode = Remap` uniquement. Ou réduire ρ_boost. |
| **Fallback** | `accelerated_forgetting = false` |

### R3 — Interaction RPE × marge × oubli → instabilité

| Attribut | Valeur |
|---|---|
| **Probabilité** | Moyenne |
| **Impact** | Modéré — oscillations d'accuracy |
| **Mécanisme** | La marge module r → r_eff change → δ change → l'oubli accéléré réagit différemment → les conductances oscillent. |
| **Détection** | Protocole 7 + visuel : sparkline d'accuracy oscillante au lieu de convergente |
| **Mitigation** | **Implémenter un seul axe à la fois** : RPE seul → tester. Puis oubli seul → tester. Puis marge seul → tester. Ne jamais activer les trois simultanément sans validation individuelle. |
| **Fallback** | Désactiver marge et/ou oubli |

### R4 — Port GPU produit des résultats numériquement divergents

| Attribut | Valeur |
|---|---|
| **Probabilité** | Élevée (différences float f32 entre CPU/GPU sont inévitables) |
| **Impact** | Faible si écart < 1%, Modéré si > 5% |
| **Mécanisme** | L'ordre des opérations atomiques (budget, readout) n'est pas déterministe sur GPU. Le jitter hash du decay peut différer. Les réductions atomiques (readout, budget) donnent des résultats qui dépendent de l'ordonnancement. |
| **Détection** | Protocole 2.1 : comparer accuracy CPU vs GPU |
| **Mitigation** | Accepter ±1% de variance. Le non-déterminisme du GPU est structurel et acceptable. Documenter l'écart mesuré. |
| **Fallback** | Aucun nécessaire si < 3% |

### R5 — Le CSR outgoing sur GPU consomme trop de mémoire à grande échelle

| Attribut | Valeur |
|---|---|
| **Probabilité** | Faible à 50k nœuds (<3 MB), Moyenne à >500k |
| **Impact** | Faible (le CSR outgoing est proportionnel au nombre d'arêtes) |
| **Mécanisme** | À 5M nœuds / 57M arêtes : CSR outgoing = 4B × 5M + 4B × 57M = ~248 MB. Acceptable pour GPU modernes (8–24 GB). |
| **Détection** | Protocole 2.3 : mesurer la mémoire GPU à 50k |
| **Mitigation** | Rien à 50k. À >500k : évaluer si la mémoire GPU est suffisante. |
| **Fallback** | Fallback CPU pour le decay adaptatif si mémoire GPU insuffisante (mode hybride) |

### R6 — L'overhead GPU des 3 nouvelles passes dépasse 15%

| Attribut | Valeur |
|---|---|
| **Probabilité** | Faible (estimation : ~7-10%) |
| **Impact** | Modéré — les benchmarks à grande échelle seront ralentis |
| **Mécanisme** | Les passes reverberation et adaptive_decay sont bound par les lectures mémoire (N + E lectures). Si le bandwidth GPU est saturé par les passes existantes, l'ajout peut être plus coûteux que prévu. |
| **Détection** | Protocole 6.1 |
| **Mitigation** | Fusionner reverberation + adaptive_decay + dissipation en un seul shader (comme apply_and_dissipate). Fusionner snapshot + readout. |
| **Fallback** | Garder les passes séparées mais avec des gates conditionnels (`if reverb_gain > 0`, `if k_local > 0`) pour skip quand désactivé |

### R7 — Le RPE empêche la convergence en mode inversé

| Attribut | Valeur |
|---|---|
| **Probabilité** | Très faible |
| **Impact** | Élevé — le résultat décisif (Δ +84 pp) est perdu |
| **Mécanisme** | Au début du mode inversé, $\bar{r} \approx -1$ → $\delta \approx +2$ pour les succès. Mais comme les succès sont rares, la baseline reste basse longtemps → δ reste fortement positif pour les succès → le renforcement devrait fonctionner. Le risque théorique est nul. |
| **Détection** | Protocole 1.2 : Δ_inversé < +77 pp |
| **Mitigation** | Aucune prévue (confiance haute). Si observé : investiguer α. |

| R2 | Conditionner à Remap only | `accelerated_forgetting = false` |
| R3 | Activer les axes un par un | Désactiver marge et/ou oubli |
| R4 | Documenter l'écart | Aucun (structurel) |
| R5 | Vérifier mémoire | Fallback CPU pour adaptive_decay |
| R6 | Fusionner les shaders | Gates conditionnels |

## 5. Risque global V5.2

**Confiance** : **Haute** (8/10).

Le RPE est un mécanisme bien compris en neurosciences computationnelles (Schultz 1997, Frémaux & Gerstner 2016). La prédiction théorique ($\delta \approx 0$ en régime → pas de renforcement aveugle) est solide. Le port GPU des fonctionnalités V5 est mécanique (pas de risque algorithmique). Les risques principaux sont paramètriques (choix de α) et non structurels.

Le seul risque non-trivial est R3 (interaction entre mécanismes), qui est mitigé par l'implémentation séquentielle (un axe à la fois).
