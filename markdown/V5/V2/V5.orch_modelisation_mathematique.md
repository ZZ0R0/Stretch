# V5.orch — Modélisation mathématique : invariants du framework unifié

> Formalisation des invariants que le refactoring doit préserver.
> Ce document prouve que le framework unifié ne modifie aucune équation.

---

## 1. Principe fondamental

Le refactoring V5.orch est **purement structurel**. Il ne modifie aucune équation mathématique du système. Ce document formalise les invariants que les tests de non-régression doivent vérifier.

---

## 2. Invariant I1 — Séquence de phases

### 2.1 Ordre canonique V5.1

Le pipeline V5.1 exécute les phases dans un ordre strict. Cet ordre est un **invariant** du système : le modifier changerait les résultats.

$$\pi_{\text{V5.1}} = (\text{Reset}, \text{Inject}, \text{Zones}, \text{SourceContribs}, \text{Propagation}, \text{Apply}, \text{Reverb}, \text{Decay}, \text{Dissipation}, \text{Plasticity}, \text{Budget}, \text{Sync}, \text{Readout}, \text{Metrics})$$

### 2.2 Invariant

$$\pi_{\text{V5.orch}} = \pi_{\text{V5.1}}$$

L'orchestrateur et les deux backends doivent exécuter les phases dans **exactement le même ordre** que V5.1.

### 2.3 Preuve par construction

L'orchestrateur appelle `backend.execute_tick(&tick_params)`. Le backend interne (CPU ou GPU) exécute les phases dans l'ordre défini par le `Phase` enum. L'enum est ordonné et l'implémentation itère du premier au dernier variant.

---

## 3. Invariant I2 — Propagation

### 3.1 Équation (inchangée)

Pour chaque nœud cible $j$ :

$$\phi_j(t) \leftarrow \phi_j(t) + \sum_{i \in N_{\text{in}}(j)} s_i(t) \cdot C_{ij}(t) \cdot w_{ij}$$

où :

$$s_i(t) = \phi_i(t) \cdot \text{sign}(i) \cdot \text{gain\_mod}(i) \cdot g_{\text{prop}}$$

### 3.2 Invariant CPU

Le `CpuBackend` appelle les mêmes fonctions :
- `propagation::compute_source_contribs()` — identique
- `propagation::compute_influences_fired_only()` — identique
- `propagation::apply_influences()` — identique

Le résultat est **bit-identical** car les mêmes buffers, les mêmes algorithmes, les mêmes paramètres sont utilisés.

### 3.3 Invariant GPU

Le `GpuBackend` construit `GpuParams` depuis `TickParams`. Les champs utilisés par les shaders `source_contribs.wgsl` et `propagation.wgsl` sont :
- `propagation_gain` ← `TickParams.propagation_gain`
- `gain_inhibitory` ← `TickParams.gain_inhibitory`

Le mapping est 1:1. Les shaders ne sont pas modifiés. Le résultat GPU est **identique** à V5.1 (aux différences de scheduling atomique GPU près, qui sont déjà présentes en V5.1).

## 4. Invariant I3 — Dissipation

### 4.1 V4 decay (mode standard)

$$\phi_i(t+1) = \phi_i(t) \cdot (1 - \alpha_{\text{eff}})$$

avec $\alpha_{\text{eff}} = \alpha_{\text{base}} \cdot (1 + j_i)$, où $j_i$ est le jitter hash du nœud $i$ au tick $t$.

### 4.2 V5 adaptive decay

$$\alpha_{\text{eff},i}(t) = \alpha_{\text{base}} \cdot \text{clamp}(1 - k_{\text{local}} \cdot \bar{a}_{N(i)}, 0.2, 1.0)$$

### 4.3 Invariant

Le `CpuBackend` appelle exactement les mêmes fonctions et le même code que `step_cpu()`. Le `GpuBackend` passe les mêmes paramètres au même shader. Aucune équation ne change.

---

## 5. Invariant I4 — Plasticité STDP

### 5.1 Équation complète (V4, inchangée)

Pour chaque arête $(i, j)$ :

$$\psi_{ij} = A^+ \cdot e^{-\frac{\Delta t}{\tau^+}} \cdot \mathbb{1}[\Delta t > 0] + A^- \cdot e^{-\frac{|\Delta t|}{\tau^-}} \cdot \mathbb{1}[\Delta t < 0]$$

$$e_{ij}(t+1) = (1 - \lambda_e) \cdot e_{ij}(t) + \psi_{ij}$$

$$\Delta C_{ij}^{\text{3f}} = g_{\text{plasticity}} \cdot d(t) \cdot e_{ij}(t)$$

$$\Delta C_{ij}^{\text{homéo}} = \rho \cdot (C_0 - C_{ij})$$

$$C_{ij}(t+1) = \text{clamp}(C_{ij}(t) + \Delta C_{ij}^{\text{3f}} + \Delta C_{ij}^{\text{homéo}},\; C_{\min},\; C_{\max})$$

### 5.2 Invariant

Le signal dopaminergique $d(t)$ est calculé dans l'orchestrateur et passé au backend via `TickParams.dopamine_level` et `TickParams.dopa_phasic`. La valeur calculée est **identique** car la séquence est préservée :

1. Orchestrateur : `dopamine_system.update(0.0, ...)` — decay de la dopamine
2. Backend : exécute plasticité avec le niveau de dopamine du `TickParams`
3. Orchestrateur (après readout) : `dopamine_system.update(reward_val, ...)` — update phasique

La séparation est identique à V5.1 :
- `step_gpu()` : decay dopamine → GPU tick (plasticity utilise dopamine courante) → readout → reward → update dopamine
- `step_cpu()` : identique

### 5.3 Point d'attention : timing de la dopamine

La dopamine utilisée pour la plasticité au tick $t$ est celle du **tick $t-1$** modifiée par le decay, **pas** celle mise à jour par le reward du tick $t$. Cet ordre est critique et **préservé** par l'orchestrateur :

$$d_{\text{plasticité}}(t) = (1 - \lambda_d) \cdot d_{\text{total}}(t-1)$$

Le reward du tick $t$ ne modifie la dopamine qu'**après** l'exécution du pipeline.

## 6. Invariant I5 — Budget synaptique

### 6.1 Équation

Pour chaque nœud source $i$ :

$$S_i = \sum_{j \in N_{\text{out}}(i)} C_{ij}$$

Si $S_i > B$ : $C_{ij} \leftarrow C_{ij} \cdot \frac{B}{S_i}$ pour tout $j$.

### 6.2 CPU vs GPU

- **CPU** : budget incrémental (C4, dirty-source tracking, séquentiel)
- **GPU** : budget global (2 passes atomiques, parallèle)

Ces deux implémentations donnent des résultats mathématiquement identiques (aux erreurs float près), mais par des chemins différents. Le refactoring ne change ni l'un ni l'autre — le `CpuBackend` appelle la même fonction STDP, le `GpuBackend` exécute les mêmes shaders.

---

## 7. Invariant I6 — Readout

### 7.1 Équation

Pour chaque classe $c$ :

$$S_c = \sum_{i \in G_c} \phi_i$$

$$\text{décision} = \arg\max_c S_c$$

### 7.2 Invariant

Le readout est calculé par le backend, mais la décision (argmax) est faite dans l'orchestrateur. En V5.1 :
- `step_gpu()` : GPU calcule $S_c$ → readback → argmax en Rust
- `step_cpu()` : `OutputReader::readout()` calcule $S_c$ + argmax en Rust

En V5.orch, le backend retourne `readout_scores: Vec<f32>` ($S_c$ pour chaque classe), et l'orchestrateur fait l'argmax. La décision est **identique**.

---

## 8. Invariant I7 — Reward et Dopamine

### 8.1 Séquence exacte (doit être préservée)

```
1. decay dopamine              : d_total *= (1 - λ_d)
2. [backend exécute le tick]
3. readout → scores
4. decision = argmax(scores)
5. reward = +1 si correct, -1 sinon
6. reward_system.set_reward(reward)
7. dopamine_system.update(reward)  : d_phasic += g_r × reward
8. spatial_dopamine update         : reward_center = centroïde du groupe
9. current_trial += 1
10. reward_system.clear()
```

### 8.2 Invariant

Cette séquence est **identique** dans `step_gpu()` et `step_cpu()`. L'orchestrateur l'implémente une seule fois dans `process_readout()`. L'étape 1 (decay) est avant `execute_tick()`, les étapes 3-10 sont après.

## 9. Résumé des invariants

| ID | Invariant | Vérification |
|---|---|---|
| I1 | Ordre des phases préservé | Inspection du code + test non-régression |
| I2 | Propagation bit-identical (CPU), identical (GPU) | Comparer l'énergie à tick 100, 1000, 10000 |
| I3 | Dissipation bit-identical (CPU), identical (GPU) | Comparer l'énergie résiduelle |
| I4 | Plasticité bit-identical (CPU), identical (GPU) | Comparer conductance moyenne |
| I5 | Budget bit-identical (CPU), identical (GPU) | Comparer max conductance |
| I6 | Readout + décision identiques | Comparer accuracy |
| I7 | Reward + dopamine timing préservé | Comparer dopamine_level, accuracy, reward_cumulative |

**Tous les invariants sont vérifiables par des tests de non-régression comparant les métriques V5.1 ↔ V5.orch.**
