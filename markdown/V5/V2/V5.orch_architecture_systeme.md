# V5.orch — Architecture système : Framework d'exécution unifié

> Plan détaillé du refactoring structurel de l'orchestrateur.
> Chaque section donne le code cible, les justifications, et les fichiers impactés.

---

## 1. Vue d'ensemble — Architecture cible

### 1.1 Avant (V5.1) : double chemin

```
Simulation::step()
    ├── match Gpu → step_gpu()
    │     ├── trial state machine   ← DUPLIQUÉ
    │     ├── build GpuParams (45+ champs)
    │     ├── gpu.run_full_tick()   ← 15 passes
    │     ├── readout → decision    ← DUPLIQUÉ
    │     ├── reward → dopamine     ← DUPLIQUÉ
    │     └── spatial dopamine      ← DUPLIQUÉ
    │
    └── match Cpu → step_cpu()
          ├── zones
          ├── stimuli
          ├── trial injection       ← DUPLIQUÉ (variante)
          ├── propagation
          ├── reverberation (V5)
          ├── dissipation
          ├── plasticity
          ├── readout → decision    ← DUPLIQUÉ
          ├── reward → dopamine     ← DUPLIQUÉ
          └── spatial dopamine      ← DUPLIQUÉ
```

### 1.2 Après (V5.orch) : orchestrateur unique

```
Simulation::step()                      ← ORCHESTRATEUR UNIQUE
    ├── 1. trial_state_machine()        → TrialContext { stimulus_class, need_readout, reset }
    ├── 2. build_tick_params()          → TickParams (source unique de paramètres)
    ├── 3. backend.execute_tick(&tick_params)  ← DISPATCH via trait
    │       ├── GpuBackend: build GpuParams from TickParams → single submit (15+ passes)
    │       └── CpuBackend: exécute phases séquentiellement (zones → propagation → ...)
    ├── 4. readout_and_decide()         → Option<Decision> (argmax des scores backend)
    ├── 5. reward_and_dopamine()        → met à jour reward, dopamine, spatial dopa
    ├── 6. advance_trial()              → current_trial += 1 si readout effectué
    └── 7. record_metrics()             → métriques + perf monitoring
```

### 1.3 Flux de données

```
┌─────────────────────┐    TickParams     ┌──────────────────┐
│   Orchestrateur     │ ───────────────→ │   ComputeBackend  │
│                     │                   │   (GPU ou CPU)    │
│ • trial timing      │                   │                   │
│ • TickParams build  │  TickResult       │ • execute phases  │
│ • decision argmax   │ ←─────────────── │ • return scores   │
│ • reward/dopamine   │                   │ • return metrics  │
│ • metrics recording │                   └──────────────────┘
└─────────────────────┘
```

## 2. Structures de données

### 2.1 `Phase` enum — source unique de vérité

```rust
/// Toutes les phases de calcul d'un tick, dans l'ordre canonique.
/// Le compilateur Rust force l'implémentation exhaustive de chaque phase
/// dans chaque backend (via match).
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum Phase {
    // --- Pré-calcul ---
    ResetActivations,     // Remise à zéro (full/partial/none)
    InjectStimulus,       // Injection I/O groupes
    
    // --- Zones PID ---
    ZoneMeasure,          // Mesure activité par zone
    ZonePid,              // Calcul PID
    ZoneApply,            // Application θ_mod + gain_mod
    
    // --- Propagation ---
    SourceContribs,       // activation × sign × gain_mod × propagation_gain
    Propagation,          // CSR incoming : accumule contribs × conductances
    ApplyInfluences,      // Application des influences aux activations
    
    // --- V5 Sustained ---
    Reverberation,        // φ_i += reverb × φ_prev_i
    AdaptiveDecay,        // α_eff = α × (1 - k × mean_neighbor)
    
    // --- Dissipation ---
    Dissipation,          // Decay standard (+ jitter V4)
    UpdateFatigueTrace,   // Fatigue, inhibition, trace, excitability
    
    // --- Plasticité ---
    Plasticity,           // STDP + éligibilité + 3 facteurs + homéostasie + consolidation
    BudgetSum,            // Budget synaptique pass 1 (somme par source)
    BudgetScale,          // Budget synaptique pass 2 (normalisation)
    SyncConductances,     // Recopie conductances → CSR
    
    // --- Readout ---
    Readout,              // Accumulation scores par output group
    
    // --- Métriques GPU ---
    MetricsReduce,        // Réduction parallèle (GPU) ou fold (CPU)
    
    // --- Snapshot ---
    SnapshotActivations,  // Copie activations → prev_activations (pour réverbération)
}
```

**19 phases**. Chaque backend doit implémenter ou ignorer (avec justification) chaque phase.

## 3. Le trait `ComputeBackend`

### 3.1 Interface

```rust
/// Trait commun CPU/GPU. Chaque backend implémente l'exécution
/// de toutes les phases d'un tick.
pub trait ComputeBackend {
    /// Exécuter un tick complet avec les paramètres donnés.
    /// L'ordre des phases est déterminé par l'orchestrateur.
    fn execute_tick(&mut self, params: &TickParams) -> TickResult;
    
    /// Lire les métriques brutes (pour le snapshot métriques).
    fn read_metrics(&self) -> RawMetrics;
    
    /// Télécharger l'état complet pour le snapshot viz (CPU-only).
    fn download_state(&self, domain: &mut Domain);
    
    /// Type du backend ("cpu" ou "gpu") pour le logging.
    fn name(&self) -> &'static str;
}
```

### 3.2 Pourquoi `execute_tick()` et pas `execute_phase()` ?

**Performance GPU** : le GPU utilise un single CommandEncoder avec un seul `queue.submit()`. Découper en `execute_phase()` forcerait un submit/readback par phase, détruisant les performances (latence ~100µs par submit × 19 phases = ~2ms d'overhead pur).

La méthode `execute_tick()` laisse le `GpuBackend` organiser son CommandEncoder librement. Le `CpuBackend` exécute les phases séquentiellement en interne.

### 3.3 Pourquoi pas un enum au lieu d'un trait ?

Un `enum { Cpu(CpuBackend), Gpu(GpuBackend) }` avec `match` serait aussi valide. Avantages du trait :
- Extensible à un futur backend (Metal, Vulkan natif, CUDA)
- `dyn ComputeBackend` permet le boxing si nécessaire

Mais l'enum est aussi acceptable et évite le dispatch dynamique. **Recommandation** : utiliser un enum interne avec dispatch statique, comme le `ComputeBackend` actuel.

```rust
pub enum Backend {
    Cpu(CpuBackend),
    Gpu(GpuBackend),
}

impl Backend {
    fn execute_tick(&mut self, params: &TickParams) -> TickResult {
        match self {
            Backend::Cpu(cpu) => cpu.execute_tick(params),
            Backend::Gpu(gpu) => gpu.execute_tick(params),
        }
    }
}
```

### 4.2 `execute_tick()`

```rust
impl GpuBackend {
    fn execute_tick(&mut self, params: &TickParams) -> TickResult {
        // 1. Convertir TickParams → GpuParams (mapping 1:1)
        let gpu_params = self.build_gpu_params(params);
        
        // 2. Exécuter le pipeline GPU existant (single submit)
        self.gpu.run_full_tick(
            &gpu_params,
            params.trial.need_readout,
            params.trial.reset_activations,
        );
        
        // 3. Lire les scores readout si nécessaire
        let readout_scores = if params.trial.need_readout {
            // Nombre de classes déterminé par le nombre de groupes de sortie
            self.gpu.read_readout_scores(self.num_classes)
        } else {
            vec![]
        };
        
        TickResult { readout_scores }
    }
    
    /// Conversion TickParams → GpuParams.
    /// C'est ici que le mapping unifié → GPU-specific se fait.
    fn build_gpu_params(&self, params: &TickParams) -> GpuParams {
        GpuParams {
            num_nodes: params.num_nodes as u32,
            num_edges: params.num_edges as u32,
            current_tick: params.tick as u32,
            propagation_gain: params.propagation_gain,
            gain_inhibitory: params.gain_inhibitory,
            activation_decay: params.activation_decay,
            activation_min: params.activation_min,
            // ... reste du mapping 1:1 ...
            stimulus_class: params.trial.stimulus_class,
            stimulus_intensity: params.stimulus_intensity,
            _pad0: 0,
            _pad1: 0,
            _pad2: 0,
        }
    }
}
```

### 4.3 Impact sur `gpu.rs`

**Aucune modification de `gpu.rs` n'est requise** pour la V5.orch. Le `GpuContext` reste tel quel. Le `GpuBackend` est un wrapper fin autour de `GpuContext` qui :
1. Convertit `TickParams` en `GpuParams`
2. Appelle les méthodes existantes de `GpuContext`
3. Lit les résultats (scores, métriques)

Les modifications de `gpu.rs` (nouveaux pipelines, GpuParams étendu) se feront dans la V5.2b.

    fn execute_tick(&mut self, params: &TickParams, domain: &mut Domain,
                    zone_manager: &mut ZoneManager, output_reader: &Option<OutputReader>,
                    input_encoder: &Option<InputEncoder>) -> TickResult {
        let config_zones = /* ... */;
        
        // Phase: ZoneMeasure + ZonePid + ZoneApply
        zone_manager.measure(domain);
        zone_manager.regulate(domain, config_zones);
        
        // Phase: InjectStimulus (avec reset policy V5)
        if params.trial.reset_activations {
            sustained::apply_reset_policy(domain, &params.reset_policy);
        }
        if params.trial.stimulus_class >= 0 {
            if let Some(ref encoder) = input_encoder {
                encoder.inject(domain, params.trial.stimulus_class as usize, params.stimulus_intensity);
            }
        }
        
        // Phase: SourceContribs + Propagation + ApplyInfluences
        propagation::compute_source_contribs(domain, /* ... */);
        propagation::compute_influences_fired_only(domain, /* ... */);
        propagation::apply_influences(domain, &self.buf_influences);
        
        // Phase: Reverberation (V5)
        if params.reverberation_enabled && params.reverb_gain > 0.0 {
            sustained::apply_reverberation(domain, &self.buf_prev_activations, params.reverb_gain);
        }
        
        // Phase: SnapshotActivations
        if params.reverberation_enabled {
            sustained::snapshot_activations(domain, &mut self.buf_prev_activations);
        }
        
        // Phase: AdaptiveDecay + Dissipation + UpdateFatigueTrace
        if params.adaptive_decay_enabled {
            sustained::apply_adaptive_decay(domain, params.activation_decay, params.k_local, params.activation_min);
        } else {
            // V4 decay + jitter (logique existante)
        }
        // Fatigue, inhibition, trace (commun V4/V5)
        // ...
        
        // Phase: Plasticity + Budget + SyncConductances
        if params.plasticity_enabled {
            stdp::update_plasticity_stdp_budget(domain, /* ... */);
        }
        domain.sync_conductances();
        
        // Phase: Readout
        let readout_scores = if params.trial.need_readout {
            if let Some(ref reader) = output_reader {
                let result = reader.readout(domain);
                result.scores
            } else { vec![] }
        } else {
            vec![]
        };
        
        TickResult { readout_scores }
    }
}

## 6. Orchestrateur refactoré

### 6.1 `Simulation::step()` — version cible

```rust
impl Simulation {
    pub fn step(&mut self) -> TickMetrics {
        let tick = self.tick;
        self.perf.begin_tick();

        // 1. Calculer le contexte du trial (UNE SEULE FOIS)
        let trial_ctx = self.compute_trial_context(tick);

        // 2. Construire TickParams (UNE SEULE FOIS)
        let tick_params = self.build_tick_params(tick, &trial_ctx);

        // 3. Mettre à jour la dopamine spatiale si nécessaire
        self.update_spatial_dopamine_if_needed();

        // 4. Exécuter le tick via le backend
        let result = self.backend.execute_tick(&tick_params);

        self.perf.end_phase(Phase::Compute);

        // 5. Dopamine decay (toujours, même sans trial)
        self.dopamine_system.update(0.0, &self.config.dopamine);

        // 6. Readout + Decision + Reward + Dopamine (si need_readout)
        if trial_ctx.need_readout {
            self.process_readout(&result, &trial_ctx);
        }

        self.reward_system.clear();
        self.perf.end_phase(Phase::ReadoutReward);

        // 7. Métriques
        self.record_metrics_if_needed(tick);
        let tick_metrics = self.latest_tick_metrics(tick);

        self.perf.end_tick(tick);
        self.tick += 1;
        self.check_finished();

        tick_metrics
    }
    
    /// Extrait la logique trial dupliquée entre step_gpu/step_cpu.
    fn compute_trial_context(&self, tick: usize) -> TrialContext {
        let mut ctx = TrialContext {
            stimulus_class: -1,
            reset_activations: false,
            need_readout: false,
            target_class: None,
            in_stimulus: false,
        };

        if self.current_trial < self.trials.len() {
            let trial = &self.trials[self.current_trial];
            if tick == trial.start_tick {
                ctx.reset_activations = true;
            }
            if tick >= trial.start_tick && tick < trial.start_tick + trial.presentation_ticks {
                ctx.stimulus_class = trial.class as i32;
                ctx.target_class = Some(trial.target_class);
                ctx.in_stimulus = true;
            }
            let read_tick = trial.start_tick + trial.presentation_ticks + trial.read_delay;
            if tick == read_tick {
                ctx.need_readout = true;
                ctx.target_class = Some(trial.target_class);
            }
        }
        ctx
    }
    
    /// Extrait la logique reward dupliquée : readout → decision → reward → dopamine.
    fn process_readout(&mut self, result: &TickResult, trial_ctx: &TrialContext) {
        if result.readout_scores.is_empty() { return; }
        
        // Argmax → decision
        let decision = result.readout_scores.iter().enumerate()
            .max_by(|(_, a), (_, b)| a.partial_cmp(b).unwrap_or(std::cmp::Ordering::Equal))
            .map(|(i, _)| i)
            .unwrap_or(0);
        self.last_decision = Some(decision);
        
        // Logging
        if self.total_evaluated > 0 && self.total_evaluated % 10 == 0 {
            eprintln!("  [t={} trial={}] target={} dec={} scores={:?}",
                self.tick, self.current_trial,
                trial_ctx.target_class.unwrap_or(99), decision,
                result.readout_scores);
        }
        
        // Reward + Dopamine
        if let Some(target) = trial_ctx.target_class {
            self.total_evaluated += 1;
            let correct = decision == target;
            if correct { self.correct_count += 1; }

            let reward_val = if correct {
                self.config.reward.reward_positive
            } else {
                self.config.reward.reward_negative
            };
            self.reward_system.set_reward(reward_val);
            self.dopamine_system.update(reward_val, &self.config.dopamine);
            
            // Spatial dopamine center update
            self.update_reward_center(correct, target, decision);
        }
        
        self.current_trial += 1;
    }
}
```

## 8. Schéma de migration

### 8.1 Étapes de refactoring (dans l'ordre)

```
Étape 1 : Extraire TrialContext
    └── Créer compute_trial_context()
    └── L'utiliser dans step_gpu() et step_cpu() (vérifier non-régression)

Étape 2 : Extraire process_readout()
    └── Créer process_readout()
    └── L'utiliser dans step_gpu() et step_cpu() (vérifier non-régression)

Étape 3 : Créer TickParams + build_tick_params()
    └── Vérifier que tous les champs de GpuParams sont couverts
    └── Vérifier que tous les paramètres de step_cpu() sont couverts

Étape 4 : Créer GpuBackend wrapper
    └── GpuBackend::execute_tick() appelle gpu.run_full_tick()
    └── Vérifier non-régression GPU

Étape 5 : Créer CpuBackend
    └── Déplacer la logique de step_cpu() vers CpuBackend::execute_tick()
    └── Vérifier non-régression CPU

Étape 6 : Unifier step() → orchestrateur
    └── Remplacer step_gpu()/step_cpu() par backend.execute_tick()
    └── Supprimer step_gpu() et step_cpu()
    └── Validation complète non-régression
```

### 8.2 Stratégie de non-régression

À chaque étape, valider :
1. `cargo build --release` → 0 erreurs, 0 warnings
2. Exécuter la config `config_v5_training.toml` (mode normal) → accuracy identique
3. Exécuter la config inversé → accuracy identique

Les étapes 1-3 sont **purement extractives** (pas de changement de comportement). Les étapes 4-6 sont **structurelles** (changement de dispatch, même comportement).
