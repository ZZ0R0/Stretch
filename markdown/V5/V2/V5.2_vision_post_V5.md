## 2. Trois axes stratégiques

### Axe A : Plasticité fiable (V5.3)

**Objectif** : faire en sorte que l'apprentissage ne dégrade *jamais* les performances, quel que soit le mode ou le backend.

**Sous-objectifs** :
1. $\Delta(\text{FL vs Topo}) \geq 0$ sur CPU vanilla (actuellement -20.3 pp)
2. $\Delta(\text{FL vs Topo}) \geq 0$ sur CPU sustained (actuellement -9.4 pp)
3. Remap accuracy > 60% après inversion

**Leviers identifiés** :
- Augmenter $\lambda$ (spatial selectivity) : 0.15 → 1.0–2.0
- Réduire $\eta$ (plasticity gain) : 3.0 → 0.5–1.0
- Reward-gated plasticity ($\delta_{\min}$ threshold)
- Dé-consolidation conditionnelle
- Learning rate scheduling ($\eta(t)$ décroissant)

**Méthode** : grid search sur ($\lambda$, $\eta$, $\rho$, $\delta_{\min}$) × 3 seeds. 192 configs, ~5 min GPU total.

### Axe B : Alignement CPU ↔ GPU (V5.3)

**Objectif** : écart CPU/GPU < 3pp en TopologyOnly (actuellement ~10pp).

**Action principale** : séparer `apply_and_dissipate.wgsl` en deux passes :
1. `apply_influences.wgsl` — accumulation des influences, seuil, activation
2. `dissipation.wgsl` — fatigue, inhibition, trace, decay

Cela aligne le timing des mises à jour entre CPU et GPU.

**Action secondaire** : documenter et accepter les divergences structurelles irréductibles (jitter hash, ordre atomique, zones vs injection).

### Axe C : Rigueur expérimentale (V5.4)

**Objectif** : résultats reproductibles avec intervalles de confiance.

**Actions** :
- Multi-seed CLI (`--seeds N`) systématique
- Rapport $\mu \pm \sigma$ pour chaque benchmark
- Courbes d'apprentissage tick-by-tick exportées
- Tests statistiques A/B (Welch's t-test) pour comparer configs

## 3. Horizon V6 : au-delà de la plasticité locale

La V5 a travaillé dans le cadre des **règles locales** (STDP + éligibilité + trois facteurs + budget). Ce cadre a ses limites fondamentales :

1. **Credit assignment** : les trois facteurs ne distinguent pas causalité et corrélation. Les nœuds simplement actifs (mais non causaux) sont modifiés autant que les nœuds responsables.

2. **Hiérarchie** : le réseau est « plat ». Il n'y a pas de couches, de modules, ni de représentations intermédiaires. L'information se propage par la topologie 3D brute.

3. **Multi-tâches** : le readout est binaire (2 classes). L'extension à $N$ classes nécessite un readout multi-sorties et potentiellement des mécanismes de contexte/attention.

### Directions V6 possibles

| Direction | Complexité | Impact potentiel |
|-----------|-----------|-----------------|
| Hiérarchie micro/méso/macro (zones fonctionnelles) | Haute | Représentations, abstraction |
| Credit assignment par contraste (contrastive Hebbian) | Modérée | Plasticité sélective sans backprop |
| Attention/gating spatialisé | Haute | Signal directionnel de l'apprentissage |
| Multi-tâches (>2 classes) | Faible | Extension fonctionnelle directe |
| Scaling 500k → 5M | Modérée | Capacité réseau, complexité des patterns |

## 4. Critères de sortie V5 → V6

La V5 est considérée comme terminée (gate V6) quand :

| Critère | Seuil | Statut actuel |
|---------|-------|---------------|
| Plasticité constructive (Normal, tous backends) | Δ ≥ 0 pp | ✅ GPU (+5.1), ❌ CPU (-9.4) |
| Inversé appris | Δ ≥ +20 pp | ✅ GPU (+26.5), ✅ CPU (+84) |
| Remap fonctionnel | > 60% | ❌ (25%) |
| GPU ≈ CPU | < 5pp écart | ❌ (~10pp) |
| Reproductibilité | σ < 3pp sur 5 seeds | Non mesuré |

**Estimation** : 2 à 3 itérations de V5.3/V5.4 nécessaires pour atteindre la gate V6.

---

## 5. Résumé

La V5 a accompli son objectif fondamental : **prouver l'apprentissage réel** et le rendre **constructif** (mode normal). Le pipeline est complet (CPU + GPU), documenté, testé.

Les problèmes restants sont **quantitatifs** (paramètres à ajuster, divergences à corriger) plutôt que **qualitatifs** (l'architecture fonctionne). C'est une base solide pour les optimisations V5.3/V5.4 et la transition vers les représentations hiérarchiques de la V6.
