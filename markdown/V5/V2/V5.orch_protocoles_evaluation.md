# V5.orch — Protocoles d'évaluation : Framework d'exécution unifié

> Plan de test pour valider que le refactoring structurel ne modifie aucun résultat.
> Chaque protocole est autonome et reproductible.

---

## Protocole O1 — Non-régression CPU (BLOQUANT)

### Objectif
Vérifier que le `CpuBackend` produit des résultats **bit-identical** à `step_cpu()` V5.1.

### Procédure

1. **Avant refactoring** : exécuter sur CPU les 4 configs de la matrice V5.1 et enregistrer les métriques finales.

```bash
# V5.1 — référence
cargo run --release -- --config configs/config_v5_training.toml --backend cpu > ref_normal_full.log 2>&1
cargo run --release -- --config configs/config_v5_training_topo.toml --backend cpu > ref_normal_topo.log 2>&1
cargo run --release -- --config configs/config_v5_training_inv.toml --backend cpu > ref_inv_full.log 2>&1
cargo run --release -- --config configs/config_v5_training_inv_topo.toml --backend cpu > ref_inv_topo.log 2>&1
```

2. **Après refactoring** : exécuter les mêmes configs avec le même seed.

3. **Comparer** pour chaque config :

| Métrique | Tolérance |
|---|---|
| `accuracy` | **0.0%** (bit-identical) |
| `cumulative_reward` | **0.0** |
| `dopamine_level` à tick N | **0.0** |
| `mean_conductance` à tick N | **0.0** |
| `global_energy` à tick 100 | **0.0** |

### Critère de réussite
Toutes les métriques sont **strictement identiques** entre V5.1 et V5.orch en mode CPU.

### Justification
Le CpuBackend appelle exactement les mêmes fonctions dans le même ordre. Si une métrique diverge, c'est qu'une transformation a été mal faite (ex : ordre des phases, paramètre oublié).

## Protocole O2 — Non-régression GPU (BLOQUANT)

### Objectif
Vérifier que le `GpuBackend` produit des résultats identiques (±ε) à `step_gpu()` V5.1.

### Procédure

Identique à O1, mais avec `--backend gpu`.

### Tolérance

| Métrique | Tolérance |
|---|---|
| `accuracy` | **±0.5%** (1 trial sur ~200 peut varier) |
| `cumulative_reward` | **±2.0** |
| `dopamine_level` | **±0.01** |
| `mean_conductance` | **±0.001** |

La tolérance GPU est non-nulle car :
- Les opérations atomiques GPU (readout, budget) ont un ordonnancement non-déterministe
- Les réductions parallèles peuvent donner des résultats float légèrement différents d'un run à l'autre

### Critère de réussite
Les métriques GPU restent dans les tolérances ci-dessus sur 3 runs consécutifs.

## Protocole O3 — Cohérence CPU ↔ GPU (BLOQUANT)

### Objectif
Vérifier que le refactoring n'a pas **élargi** l'écart CPU/GPU.

### Procédure

1. En V5.1, mesurer l'écart CPU/GPU (accuracy, energy, conductance) sur la config normal FullLearning.
2. En V5.orch, mesurer le même écart.
3. Comparer : l'écart V5.orch doit être **≤ écart V5.1**.

### Critère de réussite
$|\Delta_{\text{V5.orch}}^{\text{CPU-GPU}}| \leq |\Delta_{\text{V5.1}}^{\text{CPU-GPU}}|$ pour toutes les métriques.

## Protocole O4 — Performance CPU (BLOQUANT)

### Objectif
Vérifier que le refactoring n'ajoute pas plus de 5% d'overhead CPU.

### Procédure

1. Mesurer le temps total pour 10 000 ticks en mode CPU sur la config 50k nœuds :

```bash
# V5.1
time cargo run --release -- --config configs/config_v1_benchmark_50k.toml --backend cpu --ticks 10000

# V5.orch
time cargo run --release -- --config configs/config_v1_benchmark_50k.toml --backend cpu --ticks 10000
```

2. Reporter: `total_time_v51_cpu` et `total_time_vorch_cpu`.

### Critère de réussite

$$\frac{t_{\text{V5.orch}} - t_{\text{V5.1}}}{t_{\text{V5.1}}} \leq 5\%$$

### Justification
Le CpuBackend appelle les mêmes fonctions. L'overhead potentiel vient de :
- La construction de `TickParams` à chaque tick (~50 champs, trivial)
- L'indirection du `match Backend { ... }` (coût ~0)
- La copie de `readout_scores` (Vec allocation, ~8 floats)

Ces coûts sont négligeables comparés au calcul de propagation et plasticité.

## Protocole O5 — Performance GPU (BLOQUANT)

### Objectif
Vérifier que le refactoring n'ajoute **aucun** overhead GPU.

### Procédure

Identique à O4, mais avec `--backend gpu`.

### Critère de réussite

$$\frac{t_{\text{V5.orch}} - t_{\text{V5.1}}}{t_{\text{V5.1}}} \leq 1\%$$

La tolérance est plus stricte car le GPU path ne devrait avoir **aucun** changement côté GPU. Le seul changement est la construction de `GpuParams` depuis `TickParams` au lieu de directement depuis `config`, ce qui est un mapping trivial (~0.1µs).

## Protocole O6 — Exhaustivité du match Phase (informatif)

### Objectif
Vérifier que le compilateur force bien l'implémentation de chaque phase.

### Procédure

1. Ajouter un `Phase::TestDummy` dans l'enum.
2. `cargo build` → doit produire une erreur "non-exhaustive patterns" dans les deux backends.
3. Implémenter `TestDummy` comme no-op dans les deux backends.
4. `cargo build` → doit compiler.
5. Supprimer `Phase::TestDummy`.

### Critère de réussite
Le compilateur détecte l'absence de `TestDummy` dans les deux backends et refuse de compiler.
