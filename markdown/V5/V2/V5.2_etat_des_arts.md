## 2. Pipeline GPU — Architecture actuelle

### 2.1 Enchaînement des 15 pipelines (single submit)

```
┌─────────────────────────────────────────────────┐
│ CommandEncoder (1 seul submit par tick)          │
├─────────────────────────────────────────────────┤
│  1. reset_activations    (1 thread/nœud)        │
│  2. inject               (1 thread/nœud I/O)    │
│  3. zone_measure         (1 thread/zone)        │
│  4. zone_pid             (1 thread/zone)        │
│  5. zone_apply           (1 thread/nœud)        │
│  6. source_contribs      (1 thread/nœud)        │
│  7. propagation          (1 thread/nœud cible)  │
│  8. apply_and_dissipate  (1 thread/nœud)        │
│  9. plasticity           (1 thread/arête)       │
│ 10. budget_sum           (1 thread/arête)       │
│ 11. budget_scale         (1 thread/arête)       │
│ 12. sync_conductances    (1 thread/arête)       │
│ 13. readout              (1 thread/nœud output) │
│ 14. metrics_nodes        (1 thread/nœud)        │
│ 15. metrics_edges        (1 thread/arête)       │
└─────────────────────────────────────────────────┘
                        ↓
              1 readback : GpuMetricsRaw (64 bytes)
              + readout_scores (32 bytes)
```

### 2.3 Dispatch & Profiling

- **Workgroup size** : 256 (configurable via `compute.gpu_workgroup_size`)
- **2D dispatch** : activé quand `n_workgroups > 65535` (arêtes à grande échelle)
- **Timestamp profiling** : 24 timestamps pour 12 phases, résolution GPU native
- **Zero CPU-GPU synchronisation intermédiaire** : toutes les barrières sont internes au CommandEncoder

## 3. Divergence CPU / GPU — Analyse critique

### 3.1 Le problème central

Le pipeline GPU implémente la V4. Les fonctionnalités V5 (activité soutenue) n'existent que dans le chemin CPU.

| Fonctionnalité V5 | CPU (`step_cpu()`) | GPU (`step_gpu()`) | Écart |
|---|---|---|---|
| Decay adaptatif (k_local) | ✅ `sustained::apply_adaptive_decay` | ❌ Decay fixe dans `apply_and_dissipate.wgsl` | **CRITIQUE** |
| Réverbération locale | ✅ `sustained::apply_reverberation` | ❌ Absent | **CRITIQUE** |
| Reset policy (partial/none) | ✅ `sustained::apply_reset_policy` | ❌ Reset toujours full dans `injection.wgsl` | Modéré |
| Sustain tracking | ✅ `diagnostics::SustainTracker` | ❌ Absent | Mineur |

### 3.2 Conséquences

1. **Si backend = GPU** : les paramètres `v5_sustained` (adaptive_decay, reverberation, reset_policy) de la config sont **silencieusement ignorés**. La simulation tourne en mode V4 pur.

2. **Si backend = CPU** : toutes les fonctionnalités V5 s'appliquent, mais les performances sont dramatiquement inférieures (CPU single-threaded + rayon vs GPU massif).

3. **Les résultats V5.0/V5.1** ont été obtenus en mode `auto` (GPU si disponible). Si le GPU était disponible, les mécanismes V5 sustained n'ont **pas contribué** aux résultats mesurés (87.9%, 86.3%). La preuve d'apprentissage repose donc entièrement sur le pipeline V4 + le framework de tâches V5.

### 3.3 Autres divergences mineures

| Point | CPU | GPU |
|---|---|---|
| readout num_classes | Dynamique (OutputReader) | **Hardcodé `let num_classes = 2u;`** dans `readout.wgsl` |
| Dopamine spatiale | Pleine (precompute_spatial_dopamine) | Globale uniquement (champ `dopamine` dans GpuParams) |
| Budget normalization | Incrémentale (C4 dirty-source, séquentiel) | Globale (2 passes atomiques, parallèle) |
| Fired-only propagation | Switch CSR/fired-only selon taux (C1) | Toujours CSR incoming (pas de fired-only) |

## 4. Analyse du modèle d'apprentissage

### 4.1 Pipeline de plasticité (CPU — stdp.rs)

```
Phase A : Mise à jour last_activation_tick
Phase B : Snapshot activation ticks
Phase C :
  C1. Dopamine spatiale (precompute si centre change)
  C2. Active edges filter : |endpoints récemment actifs| ⊂ τ_window
  C3. Pour chaque arête active :
      1. STDP ψ_ij = A+ × exp(-Δt/τ+) ou -A- × exp(Δt/τ-)
      2. Eligibility : e_ij = γ × e_ij + ψ_ij
  C3. Hot edges list : |e_ij| > ε
  C4. Pour chaque hot edge :
      3. Trois facteurs : ΔC = η × δ_d_local × e_ij
      4. Homéostasie : C += ρ × (C₀ - C)
      5. Consolidation : compteur si C > θ_consol pendant N ticks
  C5. Budget incrémental : normalise par source si total > budget
```

### 4.2 Pipeline de plasticité (GPU — plasticity.wgsl)

```
1 thread par arête :
  1. Calcul ψ (STDP) si endpoints actifs dans τ_window
  2. Eligibility update : e = γ×e + ψ
  3. Early exit si |e| < 1e-6
  4. Trois facteurs : ΔC = η × dopamine_global × e
  5. Homéostasie : C += homeostatic_rate × (C₀ - C)
  6. Consolidation : compteur incrémenté si conditions remplies
  7. Clamp [C_min, C_max]
```

### 4.3 Correspondance mathématique CPU ↔ GPU

| Équation | CPU (stdp.rs) | GPU (plasticity.wgsl) | Concordance |
|---|---|---|---|
| $\psi = A^+ e^{-\Delta t / \tau^+}$ | ✅ | ✅ | ✅ Identique |
| $e_{ij} = \gamma \cdot e_{ij} + \psi$ | ✅ | ✅ | ✅ Identique |
| $\Delta C = \eta \cdot \delta_d \cdot e_{ij}$ | ✅ (δ_d local spatial) | ✅ (δ_d global seul) | ⚠️ GPU = global, CPU = spatial si λ>0 |
| $C += \rho \cdot (C_0 - C)$ | ✅ (sauf consolidated) | ✅ (sauf consolidated) | ✅ Identique |
| Budget : $C \cdot \text{scale si total} > B$ | Incrémental (C4) | 2 passes atomiques | ✅ Résultat équivalent |

**Verdict** : la plasticité est mathématiquement cohérente entre CPU et GPU, à l'exception de la dopamine spatiale (CPU uniquement si `spatial_lambda > 0`). Pour les configs V5 standards (`spatial_lambda = 0`), les résultats sont identiques.

### 4.4 Dissipation — Divergence V5

| Aspect | CPU | GPU |
|---|---|---|
| Decay de base | `node.activation *= (1 - rate)` | idem (avec jitter hash) |
| Decay adaptatif V5 | $\alpha_{\text{eff}} = \alpha \cdot (1 - k \cdot \bar{a}_{\text{voisins}})$ | ❌ Absent |
| Réverbération V5 | $\phi += r \cdot \phi_{\text{prev}}$ | ❌ Absent |
| Reset policy V5 | "full" / "partial" (×0.1) / "none" | ❌ Toujours "full" |

## 5. Analyse des résultats V5.1

### 5.1 Matrice complète

| Condition | TopologyOnly | RandomBaseline | FullLearning | Δ (FL − Topo) |
|---|---|---|---|---|
| Normal [0→0, 1→1] | 97.7% | 68.8% | 87.9% | −9.8 pp ❌ |
| Inversé [0→1, 1→0] | 2.3% | — | 86.3% | +84.0 pp ✅ |
| Remap (inv@5000) | — | — | 50.0% | — |

### 5.2 Diagnostic des problèmes

**Problème 1 — Saturation du reward** :
En mode normal, $p(\text{succès}) \approx 0.98$ → $r(t) \approx +1$ constant → $\text{dopamine phasique} \approx \text{constante}$ → renforcement aveugle → la redistribution budgétaire affaiblit les arêtes topologiquement utiles. Résultat : −9.8 pp.

**Problème 2 — Re-learning trop lent** :
Remap 50.0%. L'homéostasie ($\rho = 0.0001$) met ~10 000 ticks pour ramener $C = 3.0 → C = 1.0$. Les routes pré-renforcées dominent.

**Problème 3 — CPU/GPU divergence** :
Les fonctionnalités V5 sustained ne contribuent pas quand le GPU est actif (cas par défaut). Leur impact réel n'a jamais été mesuré isolément.

**Problème 4 — readout.wgsl hardcodé** :
`let num_classes = 2u;` → impossibilité de tester >2 classes sur GPU sans modification.

### 5.3 Ce qui fonctionne

- **Preuve d'apprentissage** : Δ = +84 pp est robuste et non ambigu
- **Framework anti-biais** : géométrie symétrique, 3 baselines, mapping inversé
- **GPU single-submit** : zéro readback intermédiaire, pipeline efficace pour le V4
- **Diagnostics** : RouteScore, D_k, CT, SustainTracker, path tracer Dijkstra
- **Visualisation** : stretch-viz complet (carte I/O, chemins, clusters, sparklines)
- **Calibration multi-échelle** : lois adaptatives implémentées (gain, decay, elig vs N)

### 6.2 Axe Résultats (Apprentissage)

| Tâche | Impact | Complexité |
|---|---|---|
| Implémenter le RPE (reward prediction error) | **Critique** | Faible (CPU-side, ~30 lignes) |

### 6.3 Priorités

```
2. Port V5 sustained → GPU (unifie CPU/GPU)
3. Fix readout num_classes
4. Oubli accéléré (si RPE insuffisant pour Remap)
5. Marge + multi-seed
6. Scaling (200k, 500k, 1M)
```

## 7. Résumé

La V5.1 dispose d'une **preuve d'apprentissage solide** (Δ +84 pp) et d'un **pipeline GPU performant** (V4), mais souffre de deux problèmes structurels :

1. **Le reward ne distingue pas le normal du surprenant** → saturation, dégradation en mode normal, re-learning lent
2. **Les fonctionnalités V5 sustained ne tournent pas sur GPU** → divergence silencieuse des backends

La V5.2 doit résoudre ces deux problèmes en implémentant le RPE et en portant les shaders V5 sur GPU, tout en maintenant la preuve d'apprentissage existante.
