# V5.2 — Aboutissement

> Bilan complet de la V5.2 : implémentation, résultats expérimentaux, bugs corrigés, état final.

---

## 1. Résumé

La V5.2 a réalisé les trois objectifs structurants de la version :

1. **Refactoring de l'orchestrateur** (V5.orch) — Extraction de `process_readout()`, élimination de la duplication CPU/GPU dans la logique métier (trial, reward, dopamine).
2. **RPE (Reward Prediction Error)** — Signal de plasticité centré sur la surprise, avec modulation par la marge et oubli accéléré.
3. **Port GPU V5 sustained** — Réverbération, decay adaptatif, et reset policy portés sur GPU via 3 nouveaux shaders, avec correction critique du snapshot timing.

Le résultat décisif post-fix : **la plasticité est devenue constructive** pour la première fois (+5.1 pp en mode normal GPU, FullLearning > TopologyOnly).

## 3. Phases d'implémentation

### Phase 0 : Orchestrateur unifié (V5.orch)

| Étape | Action | Statut |
|-------|--------|--------|
| 0.2 | Extraction `compute_trial_context()` | ✅ |
| 0.3 | Extraction `process_readout()` | ✅ |
| 0.8 | Unification dans `step()` | ✅ |
| 0.9 | Nettoyage : 0 warnings, 0 errors | ✅ |

La duplication CPU/GPU de ~92 lignes de logique métier a été éliminée. Le readout, le reward, la dopamine, et la décision sont maintenant calculés une seule fois dans l'orchestrateur, indépendamment du backend.

### Phase 1 : RPE

- `RewardSystem` enrichi avec `rpe_baseline`, `rpe_delta`, `margin_modulation`
- Calcul dans `process_readout()` : $r_{\text{eff}} = r / (1 + \beta |M|)$, puis $\delta = r_{\text{eff}} - \bar{V}$
- $\bar{V}$ mis à jour par EMA : $(1 - \alpha)\bar{V} + \alpha \cdot r_{\text{eff}}$
- `rpe_delta` transmis dans GpuParams et utilisé dans le shader `plasticity.wgsl`

### Phase 2 : Port GPU V5

3 nouveaux shaders :
- `reverberation.wgsl` (~35 lignes) : $a_i \leftarrow a_i + r_{\text{gain}} \cdot a_i^{\text{prev}}$
- `adaptive_decay.wgsl` (~50 lignes) : $\alpha_{\text{eff}} = \alpha \cdot \text{clamp}(1 - k \cdot \bar{a}_{N(i)}, 0.2, 1.0)$
- `snapshot_activations.wgsl` (~20 lignes) : copie activations → prev_activations

Pipeline GPU étendu de 12 à 15 phases (single submit).

### Phase 3 : Oubli accéléré + Multi-seed CLI

- `rho_boost` implémenté dans `stdp.rs` et `plasticity.wgsl`
- CLI : `--seeds N` pour exécution multi-seed (non utilisé dans la pratique)

## 5. Bug critique corrigé : Snapshot Timing GPU

### Description

Dans `gpu.rs::run_full_tick()`, le snapshot des activations était en Phase 12 (post-decay), alors que le CPU le prend après la réverbération et avant la dissipation.

### Cause racine

Le snapshot GPU capturait $a_i \cdot (1 - \alpha_{\text{eff}})$ au lieu de $a_i$. Avec $\alpha = 0.25$, l'énergie de réverbération GPU convergeait vers ~75% du CPU, causant un déficit d'activité systématique.

### Fix appliqué

Snapshot déplacé **entre Phase 7b (reverberation) et Phase 7c (adaptive_decay)** dans `run_full_tick()` et `run_profiled_tick()`. La constante `PHASE_NAMES` mise à jour pour refléter les 15 phases dans le nouvel ordre.

### Impact mesuré

| Config | Avant Fix | Après Fix | Δ |
|--------|-----------|-----------|---|
| Normal FullLearning | 73.4% | **86.7%** | **+13.3pp** |
| Normal TopologyOnly | 76.6% | **81.6%** | **+5.0pp** |

## 6. Instabilités résiduelles

### 6.1 Gap GPU ↔ CPU (~10pp)

Le shader fusionné `apply_and_dissipate` met à jour fatigue/inhib/trace AVANT la réverbération (Phase 7), alors que le CPU fait ces mises à jour APRÈS. Impact estimé : ~5pp. Fix possible : séparer le shader en deux passes (apply + dissipate), reporté à V5.3.

L'ordre Inject→Zones sur GPU (vs Zones→Inject sur CPU) contribue aussi mais inverser cet ordre sur GPU est contre-productif (zones mesurent post-reset, activité ~0).

### 6.2 Plasticité sur CPU encore destructrice

Sur CPU vanilla, la plasticité coûte -20.3pp. Sur CPU sustained, -9.4pp. Les sustained features atténuent mais n'éliminent pas le problème. Causes identifiées :
- $\lambda = 0.15$ : champ dopaminergique trop diffus (rayon ~6.7 unités sur domaine de 100)
- $\eta = 3.0$ : gain plastique trop élevé, amplifie le bruit
- Homéostatique : force de rappel vers $C_0 = 1.0$ efface les patterns utiles

### 6.3 Remap bloqué par consolidation (25%)

Les edges consolidés pendant la phase initiale sont permanents et immunisés contre toute modification. Le réseau est « gelé ». Nécessite un mécanisme de dé-consolidation (V5.3).

### 6.4 RPE compensatoire mais non-constructif

Le RPE protège contre les dégâts plastiques (40.2% sans RPE → 73.4% avec RPE) mais ne guide pas positivement l'apprentissage. Il agit comme un frein asymétrique, pas comme un signal directionnel.

## 9. Leçons retenues

1. **Le timing du snapshot est critique** : une seule phase de décalage dans un pipeline GPU de 15 passes peut coûter 13pp d'accuracy. Les pipelines GPU doivent être validés phase par phase contre le CPU.

2. **Le RPE est nécessaire mais insuffisant** : il empêche la catastrophe plastique (40% → 73% → 86.7% avec fix) mais ne dirige pas l'apprentissage. Il faut un signal constructif, pas juste un frein.

3. **La consolidation est un couteau à double tranchant** : elle protège les bonnes routes mais empêche toute adaptation future. Un mécanisme de dé-consolidation conditionnelle est indispensable.

4. **Les tests d'isolation systématique sont essentiels** : 11 expériences ciblées ont permis d'identifier la cause racine (snapshot timing) alors que le symptôme initial (mauvais score GPU) pointait vers mille hypothèses.

5. **Le shader fusionné est une dette technique** : `apply_and_dissipate` combine trop de logique (influences, fatigue, inhibition, trace, decay), créant des différences de timing avec le CPU qu'on ne peut pas corriger sans le séparer.
